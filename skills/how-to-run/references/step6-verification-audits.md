# Verification Audits and Unverifiable-Content Sanitization

Record-time validation (Step 3), render-time sanitization (Step 4), and the Step 6 audit passes. These are the last line of defense against hallucinated tokens and markdown/link injection from untrusted README prose — apply every rule exactly as written; never drop or reorder a rule.

## Record-time validation (Step 3 — approved unverifiable items)

- `source_file`: must match `^[A-Za-z0-9][A-Za-z0-9._/-]{0,128}$`; split on `/` and drop the entry if any segment is empty, `.`, or `..` (so `docs/../etc/passwd` cannot render as a plausible attribution).
- `source_heading`: truncate to 80 chars; strip every backtick, `<`, `>`, `\`, `&`, newline, carriage return, NUL, and Cc/Cf character; then escape every remaining `[` as `\[` and `]` as `\]` (a heading like `[Source](javascript:alert(1))` must not form a clickable link; the `&` strip prevents CommonMark numeric character references from resurrecting brackets at render time; `\` is stripped, not escaped, because stacked backslashes can produce unclosed link labels). Do not whitelist heading characters — legitimate headings contain varied punctuation.

## Render-time sanitization (Step 4 — before writing each approved item)

- Sanitize `text`: truncate to 240 characters; strip every backtick, `<`, `>`, `\`, `&`, newline, carriage return, NUL, and Cc/Cf character (the `&` strip closes the numeric-character-reference decode path in CommonMark renderers).
- Keep `[`, `]`, `(`, `)` so legitimate links render — but scan EVERY `](` in the sanitized text and reject the entry entirely if any is followed by anything other than `https://`, `http://`, or a relative-path target matching `^[^:/\s\\][^:)%\s\\]*\)` (the inner class excludes `:` so embedded schemes like `javascript:` are rejected, `%` so percent-encoded schemes are rejected on pre-decoding renderers, and `\\` as a UNC backstop); additionally split the matched path on `/` and reject the entry if any segment equals `..`.
- Reject every occurrence of `![` — the markdown image opener (inline and reference-style) is never legitimate in an attribution and can smuggle active content into previewers.
- Reject reference-style link definitions (`]` + optional whitespace + `:` + destination) whose destination contains `:` or `%` unless it begins with `http://` or `https://` (case-insensitive) — rejects `[x]: javascript:alert(1)`, `[x]: data:text/html,...`, `[x]: javascript%3Aalert(1)`. Destinations with neither `:` nor `%` pass (they cannot form a scheme even after percent-decoding).
- Entries failing any check are **removed** from the `approved-unverifiable-items` list (so `rendered_line` can never stay unset and exempt blank lines). Entries that pass store the full rendered line — including any leading list marker or indentation — into `rendered_line`.

## Step 6 audits (against the written HOW-TO-RUN.md)

On any failure, follow SKILL.md Step 6: show the correction, wait for approval, apply, re-verify. Never silently accept an ungrounded token.

- **External Services re-verification** — re-apply, from `external-services-docker.md`: §Citation Format to every `- Source: [<title>](<url>)` line; §Web-Search Recipe step 4 + §Registry Allowlist (same Host Extraction algorithm, exact match) to every image reference; §Web-Search Recipe step 5 regexes to every `-e`, `-p`, `-v`, and image line; the Snippet Templates env-var rule. Additionally: reject any bare moving label (`latest`, `stable`, `edge`, `nightly`, `canary`, `main`, `current`, `rolling`) in any image reference (the step 6 vendor-page exception is not re-checkable at read-back time); every `-v` left side must match `^[A-Za-z0-9][A-Za-z0-9._-]*-[A-Za-z0-9][A-Za-z0-9._-]*-data$` (the substituted `<project-slug>-<service-slug>-data` shape). Services the user kept via the downgrade prompt may carry catalogue-rescued port/volume values, but must still pass every sanitization regex, the allowlist match, and the tag rule — if not, surface the failing rule, and after approval rewrite the section or replace it with the Shared-cloud-no-Docker template, then re-verify.
- **Pre-Conditions Block audit** — apply `external-services-docker.md` §Pre-Conditions Block "Step 6 audit" to every per-service subsection.
- **Detector-token re-validation** — in the written file: package identifiers must match `^[A-Za-z0-9][A-Za-z0-9._+:@/-]{0,99}$` with no `://` and no empty/`.`/`..` segments after splitting on `/` or `:`; the project name must match `^[A-Za-z0-9._ -]{1,64}$` or be exactly `(unknown)`; hardware/OS and recommended-tool mentions must be canonical tokens from the detector's Task 0d / 0d2 lists; schema-bootstrap filenames in invocations must match `^[A-Za-z0-9][A-Za-z0-9._/-]{0,128}$` with no empty/`.`/`..` segments and no segment starting with `-`; config-file section names echoed anywhere must match `^:?[A-Za-z_][A-Za-z0-9_.-]{0,63}$`; config-file paths echoed anywhere must match `^[A-Za-z0-9][A-Za-z0-9._/-]{0,128}$` with no empty/`.`/`..` segments. Reject any line outside skill-generated code fences that echoes free-text prose from other docs or from the user's Step 1 corrections — EXCEPT lines matching, by full-line equality after trimming leading/trailing whitespace, the `rendered_line` of an `approved-unverifiable-items` entry. Substring matches are NOT exempt — otherwise adversarial prose containing an approved fragment could bypass the check.
- **Specific-Token Audit** — build a `grounded-tokens` set from the detector's return format: **ports** (External Services Port cells + Runtime Ports cells — re-read each Runtime Ports `<file>:<line>` citation at audit time); **paths** (Source Dependencies, Environment Setup *File*, Schema Bootstrap *File* + *Directory*, Subprojects *Path*, Repos *Path* — verbatim only, no prefix/suffix expansion: `scripts/database/` does NOT ground `scripts/database/26.10.00/`); **versions** (Runtime Version Constraints, Build System Min version, and pinned version-manager file contents re-read at audit time); **counts** (row counts of detector tables + each `Variable count` cell). Then for every specific token in prose (outside skill-generated fences):
  - **Ports:** any integer 1–65535 following `port`, `localhost:`, `127.0.0.1:`, `listens on`, `bound to`, `listening at`, or as the `:`-suffix of an `http(s)://<host>:` URL must be in the grounded set (context tokens required — `Expected result: responds within 5 seconds` must not trigger on `5`). On miss, re-read the cited source; if it differs, rewrite with the observed value or delete the port (omit the URL — never substitute a framework default).
  - **Paths:** every `<dir>/<subdir>` or `<name>.<ext>` fragment must be grounded verbatim OR still exist on disk. Escape glob metacharacters (`*`, `?`, `[`, `]`, `{`, `}`) in the fragment before globbing (they survive render-time sanitization; an unescaped `src/[a-z]/x` would spuriously match). Fragments with `/`: Glob the escaped literal. Bare filenames: Glob `**/<escaped-full-name>` as one token (never split on `.` — `Dockerfile` / `Makefile` / `Procfile` have no extension). Exclude `node_modules/`, `vendor/`, `.venv/`, `venv/`, `target/`, `dist/`, `build/`, `.git/`, `.next/`, `.nuxt/`, `coverage/`, `__pycache__/` (a hallucinated `src/utils/helpers.js` must not match a transitive dependency copy). Reject fragments failing both checks.
  - **Versions:** any `X.Y.Z`, `vX.Y`, or `X.Y+` in Prerequisites / Toolchain prose must match a grounded version verbatim or a version-manager file's current content.

  A failed token is rewritten with a grounded value or deleted in favor of hedged phrasing ("the bound port", "the latest folder", "a supported version").
- **Unverified-Count filter** — strip backticks from each line first (so `` 15 `.csproj` projects `` normalizes to `15 .csproj projects`), then scan lines outside fences against the case-insensitive regex `\b\d{1,4}\s+(?:[\w.@/_-]+\s+){0,2}(?:projects?|modules?|packages?|services?|subprojects?|repos?|files?|dependencies|jars?|binaries|microservices?|workspaces?|crates?|components?|containers?|env(?:ironment)?[\s-]*(?:vars?|files?|variables?)|configurations?|profiles?|endpoints?|schemas?|tables?)\b`. A matched count passes only when it equals (a) a detector-table row count from the grounded set, or (b) a live `Glob` count for the claimed noun. Unbacked counts are rewritten — drop the number or hedge (`"multiple"`, `"several"`).
- **Section ordering** — when the *Private registry* signal is set or *Local TLS cert* is `mkcert` and the language-level install renders at or before Environment Setup, surface the conflict to the user (no auto-rewrite — reordering is editorial).
