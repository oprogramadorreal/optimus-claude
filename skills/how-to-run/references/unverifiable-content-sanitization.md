# Unverifiable-Content Sanitization

Validation and sanitization rules for content sourced from existing docs that the detector cannot verify ("Documented but unverifiable" audit items). SKILL.md Step 3 records approved items into the `approved-unverifiable-items` list and applies §Record-time validation; Step 4 renders each entry and applies §Render-time sanitization; Step 6 exempts the stored `rendered_line` values by exact match. These rules defend the generated file against markdown/link injection from untrusted README prose — apply them exactly as written.

## Record-time validation (Step 3)

Validate `source_file` strictly against `^[A-Za-z0-9][A-Za-z0-9._/-]{0,128}$`, then split on `/` and drop the entry if any resulting segment is empty, `.`, or `..` — mirrors the submodule/sibling path validation in project-environment-detector.md Task 0b so `docs/../etc/passwd` cannot render as a plausible-looking attribution. Sanitize `source_heading`: truncate to 80 chars; strip backtick, `` ` ``, `<`, `>`, `\`, `&`, newline, carriage return, NUL, and any Cc/Cf character; then escape every remaining `[` as `\[` and every `]` as `\]` before rendering, so a heading like `[Source](javascript:alert(1))` cannot form a clickable link inside the attribution preamble `Per <source_file> "<source_heading>": ...`. `&` is stripped for the same reason it is stripped from `text` in Step 4 — CommonMark numeric-character-reference decoding would otherwise resurrect a heading like `&#x5B;docs&#x5D;(javascript:alert(1))` (no literal brackets to escape) into `[docs](javascript:alert(1))` at render time, bypassing the bracket escape entirely. `\` is stripped (not escaped) because stacked backslashes combined with bracket escapes can produce unclosed link labels that confuse downstream markdown consumers. Do not whitelist heading characters — legitimate markdown headings routinely contain `:`, `'`, `!`, `?`, `+`, `#`, en/em dashes, and other punctuation.

## Render-time sanitization (Step 4)

Sanitize `text`: truncate to 240 characters, then strip every backtick, `` ` ``, `<`, `>`, `\`, `&`, newline, carriage return, NUL, and any Unicode category Cc/Cf character. (`&` is stripped because CommonMark-conformant renderers — including VS Code's preview and most IDE markdown previewers — decode HTML numeric character references inside link destinations and inline text per CommonMark §6.5; without this strip, an approved fragment like `[docs](&#106;avascript&#58;alert(1))` would survive both the `](` URL allowlist below and the `!`-preceding-`](` image check, then decode to `<a href="javascript:alert(1)">` at render time.) Keep `[`, `]`, `(`, `)` so existing markdown links render — but scan EVERY `](` in the sanitized text and reject the entry entirely if ANY of them is followed by anything other than `https://`, `http://`, or a relative-path target matching `^[^:/\s\\][^:)%\s\\]*\)` (the `\\` exclusion in both the first-char class and the body class is a defense-in-depth backstop — `\` is also pre-stripped above, but keeping it in the regex means a future strip-list edit cannot accidentally allow `\\evil.com\share` UNC-path targets through) (first char is not `:`/`/`/whitespace AND the inner class excludes `:` so embedded schemes like `javascript:` / `data:` / `file:` / `mailto:` are rejected, AND excludes `%` so percent-encoded schemes like `javascript%3Aalert(1)` are rejected on renderers that pre-decode; additionally split the matched path on `/` and reject the entry if any segment equals `..`). Reject every occurrence of `![` in the sanitized text — that opens markdown image syntax (inline `![alt](url)` OR reference-style `![alt][ref]`), and an approved README attribution never legitimately needs an inline image (an `https://attacker/x.svg` payload would otherwise pass the URL allowlist and active-content-rendering markdown previewers could execute embedded scripts). Banning the `![` opener catches the non-empty-alt-text inline form, the empty-alt-text inline form, and the reference-style image form in one rule. Also reject the entry if the sanitized text contains a reference-style link definition — a `]` followed by optional whitespace, `:`, optional whitespace, and a non-empty destination token whose scheme is anything other than `http`/`https`. Specifically: if the destination contains `:` OR `%`, accept only when it begins with `http://` or `https://` (case-insensitive) — reject `[x]: javascript:alert(1)` (no `://`), `[x]: javascript://alert(1)` (wrong scheme), `[x]: data:text/html,...`, `[x]: vbscript:msgbox(1)`, `[x]: mailto:a@b.c`, `[x]: file:///etc/passwd`, and also reject `[x]: javascript%3Aalert(1)` (percent-encoded `:` re-decodes to `:` on renderers that pre-decode link destinations, identical to the inline `](` rule's `%`-exclusion above). Destinations containing neither `:` nor `%` (e.g., `[note]: description here` or `[ref]: ./relative/path.md`) pass — they cannot form a scheme even after percent-decoding.
